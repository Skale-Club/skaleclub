# Milestone v1.0: Xpot Technical Debt Remediation

**Status:** ✅ SHIPPED 2026-03-30
**Phases:** 1-4
**Total Plans:** 10

## Overview

Surgical cleanup of the Xpot field sales CRM codebase. Four technical debt areas addressed: error handling, route file splitting, schema organization, and context refactoring. All changes preserve existing behavior and API contracts.

## Phases

### Phase 1: Error Handling Standardization

**Goal**: All Xpot API endpoints catch errors consistently and return structured responses without crashing the server
**Depends on**: Nothing
**Plans**: 1 plan

Plans:

- [x] 01-01: Fix error middleware + install express-async-errors

**Details:**

- Fixed `throw err` crash bug in `server/app.ts` error middleware
- Added `express-async-errors` for async handler rejection catching
- Added ZodError branch → 400 with `err.flatten().fieldErrors`
- Added `res.headersSent` guard for double-response prevention
- Consistent `{ message, errors? }` response shape

### Phase 2: Route File Splitting

**Goal**: Replace the single 1,042-line xpot.ts with 10 focused domain router files while keeping all 28 routes working identically
**Depends on**: Phase 1
**Plans**: 3 plans

Plans:

- [x] 02-01: Extract middleware.ts + helpers.ts
- [x] 02-02: Create 10 domain routers + index.ts barrel
- [x] 02-03: Update routes.ts import + verify

**Details:**

- Extracted `requireXpotUser`, `requireXpotManager` into `server/routes/xpot/middleware.ts`
- Extracted `getDistanceMeters`, GHL sync helpers into `server/routes/xpot/helpers.ts`
- Created 10 domain routers: auth, dashboard, metrics, accounts, visits, opportunities, tasks, sync, place-search, admin
- Deleted 1,042-line monolith

### Phase 3: Schema Organization

**Goal**: Split the 1,000+ line shared/schema.ts into 6 domain-focused files while keeping all 64 import sites working unchanged
**Depends on**: Nothing
**Plans**: 3 plans

Plans:

- [x] 03-01: Create auth.ts, chat.ts, forms.ts
- [x] 03-02: Create cms.ts, settings.ts, sales.ts

**Details:**

- Split into: auth (3 tables), chat (4 tables), cms (5 tables), forms (2 enums + 1 table), settings (3 tables + interfaces), sales (8 enums + 11 tables)
- Barrel re-export at `shared/schema.ts` (6 lines)
- drizzle.config.ts updated to folder path
- Zero consumer changes (64 import sites unchanged)

### Phase 4: Context Refactoring

**Goal**: Replace the 729-line XpotContext with focused feature hooks that each consumer can import independently
**Depends on**: Nothing (frontend-only)
**Plans**: 3 plans

## Milestone Summary

**Key Decisions:**

- Surgical refactoring approach — minimized risk by preserving all API contracts and behavior
- GeoContext for shared state — resolved useState-per-call isolation bug
- Barrel re-export pattern — enabled zero-consumer-change schema migration
- express-async-errors — lightweight solution for async error catching

**Issues Resolved:**

- Fixed crash bug: `throw err` after `res.json()` in error middleware
- Fixed 12+ uncaught ZodErrors from async route handlers
- Eliminated 1,042-line route monolith
- Eliminated 1,004-line schema monolith
- Eliminated 729-line context monolith

**Technical Debt Cleared:**

- DEBT-01: Route file size (1,042 → 13 focused files)
- DEBT-02: Schema bloat (1,004 → 6 domain files)
- DEBT-03: Context bloat (729 → 8 focused hooks)
- DEBT-04: Error handling inconsistency
