## v1.6 Skale Hub Weekly Live Gate

### Phase 25: Foundation
**Goal**: The database schema, shared contracts, and storage layer encode Skale Hub lives, participants, registrations, and access events with phone-first identity matching.
**Depends on**: Nothing (first phase of v1.6)
**Requirements**: HUB-01, HUB-02, HUB-03, HUB-04
**Plans:** 1/1 plans complete
**Completed:** 2026-05-02
**Success Criteria** (what must be TRUE):
  1. An additive migration creates `hub_lives`, `hub_participants`, `hub_registrations`, and `hub_access_events` without schema errors.
  2. `shared/schema/hub.ts` exports typed Drizzle/Zod contracts through `#shared/schema.js`.
  3. `server/storage.ts` exposes typed methods for lives, participants, registrations, and access-event summaries.
  4. Participant identity stores raw + normalized phone/email values and registration uniqueness is enforced by `(live_id, participant_id)`.
Plans:
- [x] `25-01-PLAN.md` - migration, schema module, storage foundation, phone-first identity behavior (HUB-01, HUB-02, HUB-03, HUB-04)

### Phase 26: API & Tracking
**Goal**: The server exposes public and admin Skale Hub APIs for active live discovery, registration, unlock/access tracking, and live management.
**Depends on**: Phase 25
**Requirements**: HUB-05, HUB-06, HUB-07, HUB-08, HUB-09
**Plans:** 1/1 plans complete
**Completed:** 2026-05-02
**Success Criteria** (what must be TRUE):
  1. `GET /api/skale-hub/active` returns the active live or a safe empty payload.
  2. `POST /api/skale-hub/register` upserts the participant and returns a successful unlocked response for the current live.
  3. `POST /api/skale-hub/:liveId/access` records unlock/access events distinct from registration.
  4. Admin live APIs support create, update, detail, list, and only one live can be active at a time.
  5. Admin analytics APIs return dashboard, per-live, and per-participant summary data.
Plans:
- [x] `26-01-PLAN.md` - route module, active live endpoint, registration/access tracking, admin live + summary APIs (HUB-05, HUB-06, HUB-07, HUB-08, HUB-09)

### Phase 27: Public Experience
**Goal**: Visitors get a branded Skale Hub page inside the current site, can register quickly, unlock the live link, and see a graceful empty state when no live is active.
**Depends on**: Phase 26
**Requirements**: HUB-10, HUB-11, HUB-12, HUB-13
**Plans:** 1/1 plans complete
**Completed:** 2026-05-02
**Success Criteria** (what must be TRUE):
  1. The site exposes an integrated `Skale Hub` public route using the existing layout and brand system.
  2. The page shows title, subtitle, current live card, form, unlock CTA, and unlocked link area.
  3. Successful registration shows `Registration confirmed. You can now access the live.` and enables the live link in a new tab.
  4. When no active live exists, visitors see a friendly empty state instead of a broken gate.
Plans:
- [x] `27-01-PLAN.md` - route integration, page UI, registration form, unlock state, empty state (HUB-10, HUB-11, HUB-12, HUB-13)

### Phase 28: Admin Management
**Goal**: Admins manage Skale Hub lives from the existing dashboard without leaving the product shell.
**Depends on**: Phase 26
**Requirements**: HUB-14, HUB-15
**Plans:** 1/1 plans complete
**Completed:** 2026-05-02
**Success Criteria** (what must be TRUE):
  1. Admin dashboard includes a `Skale Hub` section in the current navigation pattern.
  2. Admin can create and edit lives with title, description, date, time, external link, and status.
  3. Admin can activate one live at a time and change older lives to inactive or ended.
Plans:
- [x] `28-01-PLAN.md` - admin section wiring, live list, live editor, status/activation controls (HUB-14, HUB-15)

### Phase 29: Analytics & Reporting
**Goal**: Admin can understand overall Skale Hub participation, live-level performance, and participant history from one integrated dashboard area.
**Depends on**: Phases 26 and 28
**Requirements**: HUB-16, HUB-17, HUB-18
**Plans:** 1/1 plans complete
**Completed:** 2026-05-02
**Success Criteria** (what must be TRUE):
  1. Summary cards show total participants, total registrations, and live-by-live counts.
  2. Per-participant views show contact info, registration date, lives accessed count, last live accessed, and last access date.
  3. Per-live views show title, live date, registrations count, unlock/access counts, and participant details.
  4. Manual QA confirms duplicate participant matching, active-live behavior, and desktop/mobile usability.
Plans:
- [x] `29-01-PLAN.md` - summary dashboards, participant history table, per-live analytics, QA checklist (HUB-16, HUB-17, HUB-18)
