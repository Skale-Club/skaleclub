## Summary

Phase 7 creates `server/routes/estimates.ts` with five Express route handlers and registers them in `server/routes.ts`. All five storage methods from Phase 6 are confirmed in place and typed correctly. The route file pattern is well-established and fully consistent across all existing domain routers — the estimates file can be written by direct analogy to `portfolio.ts` and `forms.ts`.

One **critical correction** was found during research: CONTEXT.md decision D-12 states `registerEstimatesRoutes(app: Express, requireAdmin: ...)` but every existing route file in the codebase uses `registerXxxRoutes(app: Express)` only — `requireAdmin` is imported directly from `./_shared.ts`, not passed as a parameter. The `storageAdapter.ts` is the sole exception (legacy pattern). The planner MUST use the `(app: Express)` signature with a direct import, not the parameter-passing variant.

`express-async-errors` is confirmed registered in `server/app.ts` (first import). This means route handlers do NOT need try-catch — thrown errors are forwarded to the global error handler automatically. However, all existing domain route files still use explicit try-catch blocks per current convention. The planner should follow that convention for consistency.

**Primary recommendation:** Create `server/routes/estimates.ts` following the `portfolio.ts` structure exactly — import `requireAdmin` from `./_shared.ts`, use `safeParse` for POST/PUT validation, return 201 on create, return 404 JSON when record not found, no try-catch needed but match existing convention by including it.

### Pattern 5: Slug Generation for POST

`crypto` is already imported in `server/routes.ts` (line 6). In the new route file, import it and call `crypto.randomUUID()` before passing data to `createEstimate`.


### Anti-Patterns to Avoid

- **Passing `requireAdmin` as a function parameter:** All other route files import it directly. Using a parameter would diverge from the established pattern and require changing the call site in `routes.ts`.
- **Including `slug` in the POST body schema:** Slug is server-generated. Strip it before passing to `insertEstimateSchema` or use `.omit({ slug: true })` on the schema for POST validation (since `insertEstimateSchema` includes `slug`).
- **Using `setPublicCache` on the public slug endpoint:** Estimate data includes client names and pricing — no CDN caching (D-15).
- **Returning 204 on DELETE:** Existing pattern in the codebase returns `res.json({ success: true })` (portfolio.ts line 233). Forms.ts uses `res.status(204).end()` for delete. Either is acceptable — follow portfolio.ts for consistency since it's the closer analog.

## Common Pitfalls

### Pitfall 1: Slug in insertEstimateSchema

**What goes wrong:** `insertEstimateSchema` has a required `slug` field. If the POST handler validates `req.body` with it directly, validation will fail because the client never sends a slug.

**Why it happens:** The schema was designed for storage-layer inserts (which include slug), not for the HTTP request body (which should omit slug).

**How to avoid:** Use `insertEstimateSchema.omit({ slug: true })` for POST body validation, then attach the server-generated slug before calling `storage.createEstimate()`.

**Warning signs:** `ZodError: Required` on the `slug` field when testing POST.

### Pitfall 2: Route Order — slug/:slug vs :id

**What goes wrong:** If `GET /api/estimates/:id` is registered before `GET /api/estimates/slug/:slug`, Express may match `slug` as the `:id` parameter.

**Why it happens:** Express matches routes in registration order. The string `"slug"` satisfies `:id`.

**How to avoid:** Register `/api/estimates/slug/:slug` before `/api/estimates/:id`, or use separate path prefixes (the `slug/:slug` path naturally avoids collision since it has a fixed prefix segment).

**Warning signs:** 404 or NaN id parse when hitting `GET /api/estimates/slug/some-uuid`.

**Note:** In this phase, there is no `GET /api/estimates/:id` route — only `PUT` and `DELETE` by id, and `GET /api/estimates/slug/:slug`. The collision risk exists only if a future `GET /:id` is added. Still register slug route first as defensive practice.

### Pitfall 3: `updateEstimate` returns `undefined` when row not found

**What goes wrong:** `storage.updateEstimate(id, data)` destructures `[updated]` from `.returning()`. If the id doesn't exist, `updated` will be `undefined`, and the handler will return `undefined` as JSON.

**Why it happens:** Drizzle `UPDATE ... RETURNING` returns an empty array when no row matches — destructuring gives `undefined`.

**How to avoid:** After calling `updateEstimate`, check if the result is defined. Or check existence first with `storage.getEstimate(id)` before updating. The safer pattern is to check existence first (consistent with the 404 pattern from D-18).

**Warning signs:** `PUT /api/estimates/9999` returns 200 with `undefined` body.

### Pitfall 4: try-catch vs express-async-errors

**What goes wrong:** Omitting try-catch and relying solely on `express-async-errors` works, but is inconsistent with every existing domain route file (which all use explicit try-catch).

**Why it happens:** `express-async-errors` is present but existing domain files predate or ignore it for per-route handling.

**How to avoid:** Match the existing convention — include try-catch blocks in each handler. The outer catch returns the error message as JSON with 400 or 500 status.

```typescript
app.post("/api/estimates", requireAdmin, async (req, res) => {
  try {
    const bodySchema = insertEstimateSchema.omit({ slug: true });
    const parsed = bodySchema.safeParse(req.body);
```
