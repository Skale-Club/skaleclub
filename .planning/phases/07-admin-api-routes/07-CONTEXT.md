# Phase 7: Admin API Routes - Context

**Gathered:** 2026-04-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Create five Express route handlers in `server/routes/estimates.ts` and register them in `server/routes.ts`. Four endpoints require admin auth; one is public. No new database columns — depends entirely on Phase 6 storage methods.

Endpoints:
- `GET /api/estimates` — list all estimates (admin)
- `POST /api/estimates` — create estimate with server-generated UUID slug (admin)
- `PUT /api/estimates/:id` — update estimate (admin)
- `DELETE /api/estimates/:id` — delete estimate (admin)
- `GET /api/estimates/slug/:slug` — fetch by slug (public, no auth)

</domain>

<decisions>
## Implementation Decisions

### Slug Generation
- **D-01:** Slug is always auto-generated by the server using `crypto.randomUUID()` on POST — admin never provides or edits the slug
- **D-02:** Slug is immutable after creation — PUT endpoint does not accept a `slug` field (strip it from the update body or ignore it)
- **D-03:** Phase 8 UI has no slug input field — copy-link button just reads the server-returned slug

### POST Body Shape
- **D-04:** `POST /api/estimates` accepts `{ clientName: string, note?: string, services: EstimateServiceItem[] }` — single atomic create, no two-step flow
- **D-05:** `services` defaults to `[]` on create (admin can create an estimate with no services yet and add them via PUT)

### PUT Body Shape
- **D-06:** `PUT /api/estimates/:id` accepts `{ clientName?: string, note?: string, services?: EstimateServiceItem[] }` — all fields optional (partial update)
- **D-07:** `slug` is excluded from the update schema — even if sent, it is ignored

### Response Shapes
- **D-08:** All admin write endpoints (`POST`, `PUT`) return the full estimate record (all columns) — no partial returns
- **D-09:** `GET /api/estimates` list returns full estimate records — Phase 8 can pick what to display
- **D-10:** `GET /api/estimates/slug/:slug` returns the full estimate record (public viewer needs all fields)

### Route File Structure
- **D-11:** New file `server/routes/estimates.ts` — follows the domain-router pattern from `server/routes/portfolio.ts` and `server/routes/forms.ts`
- **D-12:** Export `registerEstimatesRoutes(app: Express, requireAdmin: ...) → void` — same signature as other route files
- **D-13:** Register in `server/routes.ts` alongside other domain routes

### Auth & Caching
- **D-14:** Four admin endpoints use `requireAdmin` middleware imported from `./_shared.ts`
- **D-15:** Public slug endpoint (`GET /api/estimates/slug/:slug`) does NOT use `setPublicCache` — estimate data is sensitive (client name, pricing); no CDN caching

### Validation
- **D-16:** Use `insertEstimateSchema` from `shared/schema/estimates.ts` for POST body validation
- **D-17:** For PUT, use `insertEstimateSchema.partial().omit({ slug: true })` — partial, slug excluded
- **D-18:** Return 404 JSON `{ message: "Estimate not found" }` when id/slug lookup returns undefined

### Claude's Discretion
- Import ordering within the new route file
- Error message wording for validation failures
- Whether to wrap routes in try-catch vs rely on express-async-errors (follow existing pattern — `express-async-errors` is registered, so no explicit try-catch needed per the v1.0 convention)

</decisions>
