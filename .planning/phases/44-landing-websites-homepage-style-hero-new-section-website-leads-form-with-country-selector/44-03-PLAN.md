**Single small edit:** widen the `FormQuestionType` union at shared/schema/forms.ts:189 to include `'phoneCountry'`.

Before:
```ts
export type FormQuestionType = 'text' | 'email' | 'tel' | 'select';
```

After:
```ts
export type FormQuestionType = 'text' | 'email' | 'tel' | 'select' | 'phoneCountry';
```

That is the ONLY edit to this file. The `FormQuestion` interface (lines 204-214) references `FormQuestionType` so it picks up the new value automatically. The `FormConfig` type, the table columns, `insertFormSchema`, and `formLeadProgressSchema` are all unaffected because the form question shape is stored as `jsonb config` (free-form on the server, validated only client-side at the type level).

**Why no Zod schema is needed for the new type:** The existing `insertFormSchema` (lines 233-242) treats `config` as a `z.custom<FormConfig>()` — there's no per-question-type runtime validation. The TypeScript union is the only enforcement, and it's compile-time. The seed in 44-04 will construct a typed `FormConfig` object that includes a question of `type: 'phoneCountry'`, satisfying TypeScript.

**Verification rationale:** `npm run check` will catch any place in the codebase that exhaustively switches on `FormQuestionType` — we should NOT find any new errors because the only consumer is the renderer in LeadFormModal.tsx (handled in Task 2), and that file currently uses string-equality checks (`if (currentQuestion.type === "tel")`), which are not exhaustive.
