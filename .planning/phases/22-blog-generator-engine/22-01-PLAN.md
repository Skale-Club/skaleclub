Create `server/lib/blog-generator.ts` and export `class BlogGenerator` with this runtime contract:

```typescript
type BlogGeneratorResult =
  | { skipped: true; reason: "no_settings" | "disabled" | "posts_per_day_zero" | "too_soon" | "locked" }
  | { skipped: false; reason: null; jobId: number; postId: number; post: BlogPost };
```

Implement `static async generate({ manual }: { manual: boolean })` as the single public entry point.

Preflight rules:
- load settings from `storage.getBlogSettings()`
- if no row: return `{ skipped: true, reason: "no_settings" }`
- if `manual === false` and `enabled === false`: return `{ skipped: true, reason: "disabled" }`
- if `manual === false` and `postsPerDay <= 0`: return `{ skipped: true, reason: "posts_per_day_zero" }`
- if `manual === false` and `lastRunAt` is within `24h / postsPerDay`: return `{ skipped: true, reason: "too_soon" }`

Lock rules:
- acquire the lock with a single guarded DB update on `blog_settings.lock_acquired_at`
- allow stale-lock takeover only when the old timestamp is older than 10 minutes
- if the update touches zero rows, return `{ skipped: true, reason: "locked" }`

Job lifecycle rules:
- after lock acquisition, create a generation job with `status: "running"` and `startedAt: new Date()`
- structure the file so the yet-to-be-implemented pipeline work is delegated to a private helper such as `runPipeline(...)`; Plan 02 will replace the placeholder with the real Gemini + post creation logic
- if pipeline execution throws after the job exists, call `storage.updateBlogGenerationJob(job.id, { status: "failed", error: message, completedAt: new Date() })`, clear `lockAcquiredAt`, and rethrow

Implementation detail for testability:
- keep the public API as `BlogGenerator.generate({ manual })`
- use a small internal dependency factory or optional module-level helper hooks so `server/lib/__tests__/blog-generator.test.ts` can drive skip and lock branches without talking to the live database or Gemini API

Do not implement the real topic/content/image/post pipeline in this plan. This plan ends once skip logic, lock behavior, and running-job cleanup are correct and the executable spec passes.
