<objective>
Wire the attribution layer into the lead-creation critical path so that every successful
`form_leads` insert ALSO writes an `attribution_conversions` row of type `lead_created`,
linked to the visitor's session.

The flow is:

```
Visitor submits form
  → client sends { ...formData, __visitorId: "uuid-from-localStorage-mvp_vid" }
  → server inserts form_leads row (existing code path, unchanged)
  → server runs runLeadPostProcessing(lead, config, name, visitorUuid)
     ├── (existing) Twilio notification
     ├── (existing) GHL sync
     └── (NEW) Attribution IIFE:
         await storage.linkLeadToVisitor(lead.id, visitorUuid) → int FK | null
         if FK !== null:
           await storage.createAttributionConversion({ visitorId: FK, leadId: lead.id,
             conversionType: 'lead_created', ft/lt copied from session row })
```

Two key invariants:

1. **NEVER block the lead-create critical path.** The IIFE is `try { ... } catch { log }`
   — no throw, no await on the caller side beyond the storage call itself. Even if the
   `visitor_sessions` row is missing or attribution_conversions insert fails, the user
   sees their lead succeed.
2. **`__visitorId` is the SINGLE source of truth.** We use the request BODY (not an
   `X-Visitor-Id` header) per CONTEXT.md Decision: "Lead-creation hook ... carry a
   visitor UUID (the client tracking hook puts it in the form payload as `__visitorId`
   — or use a header like `X-Visitor-Id` — pick one and keep it consistent with the
   client side)." Body field is chosen because:
   - The existing `progressUrl` POST already JSON-marshals the payload — one field add
     is cleaner than a custom header.
   - sendBeacon (which the public-site attribution tracker uses) supports custom
     headers awkwardly; the lead form's `fetch` doesn't have that concern but staying
     consistent with the rest of the form payload keeps the wire format readable.

Output: every form submission with a present `mvp_vid` produces 1 row in
`form_leads` (existing) + 1 row in `attribution_conversions` (new) + a non-null
`form_leads.visitor_id` FK.
</objective>

```ts
  // 3) Marketing attribution (Phase 45) — fire-and-forget.
  // Wrapped so any failure (missing visitor, network blip, FK violation, etc.) cannot
  // bubble up and break the lead-create caller. Logs to console but always returns.
  if (visitorUuid) {
    try {
      const fk = await storage.linkLeadToVisitor(lead.id, visitorUuid);
      if (fk !== null) {
        // Denormalize ft_*/lt_* by fetching the session row (single round-trip).
        const [session] = await db
          .select({
            ftSource: visitorSessions.ftSource,
            ftMedium: visitorSessions.ftMedium,
            ftCampaign: visitorSessions.ftCampaign,
            ftLandingPage: visitorSessions.ftLandingPage,
            ltSource: visitorSessions.ltSource,
            ltMedium: visitorSessions.ltMedium,
            ltCampaign: visitorSessions.ltCampaign,
            ltLandingPage: visitorSessions.ltLandingPage,
          })
          .from(visitorSessions)
          .where(eq(visitorSessions.id, fk));

        await storage.createAttributionConversion({
          visitorId: fk,
          leadId: lead.id,
          conversionType: 'lead_created',
          pagePath: lead.urlOrigem ?? null,
          ftSource: session?.ftSource ?? null,
          ftMedium: session?.ftMedium ?? null,
          ftCampaign: session?.ftCampaign ?? null,
          ftLandingPage: session?.ftLandingPage ?? null,
          ltSource: session?.ltSource ?? null,
          ltMedium: session?.ltMedium ?? null,
          ltCampaign: session?.ltCampaign ?? null,
          ltLandingPage: session?.ltLandingPage ?? null,
        });
      }
    } catch (err) {
      console.error('[attribution] lead-creation hook failed:', err);
      // Swallow — attribution must NEVER block the lead-create critical path.
    }
  }
```

**Why this fetches the session row** (vs. trusting the client to send ft/lt):
because the SERVER's view of ft/lt (after upsert) is canonical. The client could send
stale or absent FT/LT. The session is the source of truth — same pattern as
`server/routes/attribution.ts` conversion handler from plan 03.
