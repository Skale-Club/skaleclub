<objective>
## Goal
Let admins scope the Leads list and Dashboard metrics by form. Both sections gain a "Form" selector (default "All forms"); lead rows/detail show a form badge.

## Purpose
M3-03 shipped multi-form capture (`/f/:slug` + per-chat slug). Leads from different forms now coexist in `form_leads` but the admin UI still aggregates them blindly. Without per-form scoping, a customer running two distinct forms (e.g. "Contact" and "Newsletter") can't read the funnel accurately. This plan fixes that — last substantive slice before M3-05 cleanup.

## Output
- `GET /api/form-leads?formId=N` — server filters by form
- Leads section: form dropdown + badge on rows/detail sheet
- Dashboard: form dropdown that scopes all lead-derived metrics
- PT translations for the new strings
</objective>

<acceptance_criteria>

## AC-1: Backend filters leads by formId
```gherkin
Given two active forms (A id=1, B id=2) each with at least one lead
When admin calls GET /api/form-leads?formId=1
Then response contains only leads with form_id=1 and all existing filters (status, classificacao, completionStatus, search) still compose
```

## AC-2: Leads UI scopes list and badges rows
```gherkin
Given admin is on the Leads section with ≥2 active forms seeded
When admin opens the "Form" dropdown and picks form B
Then the leads table, the stats strip, and the detail sheet only reflect form B's leads
And every lead row displays a small badge with its form name
```

## AC-3: Dashboard metrics scope by form
```gherkin
Given admin is on the Dashboard with ≥2 active forms seeded
When admin picks form A in the "Form" dropdown (default "All forms")
Then "Total Leads", "Leads today", "Last 7 days", the funnel stages, and the hot/warm/cold breakdown only count leads where form_id matches form A
And picking "All forms" restores aggregated counts (current behavior)
```

## AC-4: Zero regression for single-form operators
```gherkin
Given a workspace that only has the default form (one form in forms table)
When admin visits Leads and Dashboard without touching the filter
Then counts and rows are identical to pre-M3-04 behavior (no empty state, no missing leads, no badge clutter beyond the default form name)
```
