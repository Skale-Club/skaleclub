**Milestone:** 3 — Multi-Forms Support
**Phase:** 2 — Admin Forms List + Editor (Phase 5.2 of the detailed plan)
**Status:** Ready for apply
**Reference:** `.planning/phases/05-multi-forms/PLAN.md` (full 5-phase roadmap)

---

## Goal

Expose the multi-forms capability to admins via a new top-level "Forms" sidebar section. Admins can list, create, rename, duplicate, archive (soft-delete), and set-default forms. The existing question editor is rewired to operate on a specific `formId` instead of the global `/api/form-config` endpoint.

After this phase ships, an admin can create a second form end-to-end (name + questions + thresholds) and archive forms they no longer need. Public-facing capture (website modal, chat widget) still routes to the default form — that comes in M3-03.

---

## Scope

**In:**

*Backend*
- `GET  /api/forms` — list forms (admin; `?includeInactive=true` to show archived)
- `POST /api/forms` — create form (validated via `insertFormSchema`)
- `GET  /api/forms/:id` — read one form (admin)
- `PUT  /api/forms/:id` — update form (validated via `updateFormSchema`)
- `DELETE /api/forms/:id` — soft-delete (hard-delete only if zero leads AND not default)
- `POST /api/forms/:id/duplicate` — duplicate form (body: `{ slug?, name? }`)
- `POST /api/forms/:id/set-default` — promote to default (transactional swap)
- Public counterpart (read-only, by slug): `GET /api/forms/slug/:slug/config` — public JSON (for M3-03 consumption; mounted here but unused until M3-03)

*Admin UI*
- New sidebar entry `forms` (top-level, separate from Leads)
- New `<FormsListSection>` component: table of forms with columns Name / Slug / Default / Active / Questions / Leads / Actions
- Per-row actions: Edit (opens editor), Duplicate, Set default, Archive/Restore
- Row-level "Create form" button (opens a small dialog: name + slug)
- `<FormEditorContent>` rewired: accepts `formId` prop, fetches `/api/forms/:id`, saves to `PUT /api/forms/:id`
- Admin routing: `/admin/forms` (list) and `/admin/forms/:id` (editor)

*Translations*
- PT strings for every new user-facing English string (per `feedback_translations.md` memory rule)

**Out (deferred):**

- Public form submission routed by slug (M3-03)
- Chat widget form selector in Chat settings (M3-03)
- Leads filter by form (M3-04)
- Dropping legacy `company_settings.form_config` column (M3-05)
- GHL field collision UI hint (deferred — permitted silently per the decision log)

## API Contracts

All admin endpoints require `requireAdmin` middleware. Responses are JSON.

### `GET /api/forms`

```
Query: ?includeInactive=true (optional, default false)
200:   [{ id, slug, name, description, isDefault, isActive, config, createdAt, updatedAt, _leadCount }]
```

Returns forms in display order (default first, then alphabetical by name). `_leadCount` is computed via `countLeadsForForm` so the UI can show it inline.

### `POST /api/forms`

```
Body: { slug, name, description?, isDefault?, isActive?, config }
201:  { ...form }
400:  Zod validation error
409:  Slug already exists
```

`config` is required; the UI seeds a sensible starter config (empty questions array, default thresholds) if the admin clicks "Create form" without customizing.

### `PUT /api/forms/:id`

```
Body: Partial<InsertFormInput>
200:  { ...form }
404:  Form not found
400:  Zod validation error
409:  Slug conflict
```

If `isDefault: true` is set, storage layer demotes the prior default in a single transaction (already implemented in `updateForm`).

### `DELETE /api/forms/:id`

```
Query: ?force=true (hard delete; requires zero leads)
204:  No content
409:  Form has leads (hint: archive instead) — when force=true
400:  Cannot delete default form
```

Default path is soft-delete (`isActive=false`). Hard delete only when admin explicitly passes `?force=true` AND `countLeadsForForm(id) === 0`.

### `POST /api/forms/:id/duplicate`

```
Body: { slug?, name? } (optional overrides)
201:  { ...newForm }
```

Creates a copy of the form's config with a unique slug (auto-generated as `<source>-copy`, `<source>-copy-2`, etc., if not provided).

### `POST /api/forms/:id/set-default`

```
200:  { ...form } (now isDefault=true, isActive=true)
404:  Form not found
```

### `GET /api/forms/slug/:slug/config` (public, for M3-03)

```
200:  { questions, maxScore, thresholds } (no metadata, no admin fields)
404:  Slug unknown or form is inactive
```

Mounted in this phase so the public schema is ready, but no frontend caller yet.

## UI Design

### FormsListSection

```
┌ Forms ────────────────────────── [+ New form] ┐
│ Create, edit, duplicate, or archive your lead │
│ capture forms.                                │
├───────────────────────────────────────────────┤
│ Name        Slug       Default Active Q  Leads│
│ Default     default    ★      On    13  14  ⋯│
│ Contact     contact-us —      On     3   0  ⋯│
│ Partners    partners   —      Off    8   2  ⋯│
└───────────────────────────────────────────────┘
```

- "Default" badge is the existing star icon style
- Actions menu (⋯): Edit, Duplicate, Set as default, Archive (or Restore if inactive), Delete (disabled if leadCount > 0, also disabled on default)
- Clicking a row name opens `/admin/forms/:id`
- Archived forms hidden by default; toggle "Show archived" in header reveals them

### Create-form dialog

```
┌ New form ─────────────────────── ✕ ┐
│ Name: [ ______________________ ]   │
│ Slug: [ ______________________ ]   │
│   (auto-filled from name)          │
│ Description (optional):            │
│ [ ____________________________ ]   │
│                                    │
│          [ Cancel ] [ Create ]     │
└────────────────────────────────────┘
```

Slug auto-suggests from name (kebab-case), editable. On create → navigate to `/admin/forms/:id` and start with empty questions + default thresholds.

### Editor (FormEditorContent rewired)

- Loads form via `useQuery(['form', formId])` → `GET /api/forms/:id`
- All saves go to `PUT /api/forms/:id` (instead of `PUT /api/form-config`)
- Top of editor adds a breadcrumb-style header: `← Forms / <form name>`
- Threshold editor unchanged — just bound to this form's config
- "Set as default" button visible on the editor header for non-default forms

## Verification / Exit Criteria

- `npm run check` passes
- `npm run build` passes
- Admin can reach `/admin/forms` from the sidebar
- Creating a form lands on its editor at `/admin/forms/:id`
- Editing questions in the new form saves independently (verified by opening the default form and seeing its questions untouched)
- Duplicating a form creates a new row with `-copy` suffix and identical config
- "Set as default" promotes the target and demotes the prior default in one round-trip
- Archive hides the row behind "Show archived"; Restore brings it back
- Delete button is disabled on default and on any form with `leadCount > 0`
- Public forms (the website modal) and chat still go through the default form (compat shim from M3-01 unchanged)
- All new UI strings have PT translations registered

## Risk Log

| Risk | Mitigation |
|---|---|
| Admin accidentally sets wrong form as default | Confirm dialog on "Set as default"; show which form is currently default in the dialog |
| Slug collision when user types an existing one | Server returns 409; UI shows inline error on the slug field |
| Duplicating a huge form (many questions) | No special handling — JSON copy is cheap; config JSON typically <50KB |
| Deleting a form referenced by a future conversation | Soft-delete preserves the row; `form_leads.form_id` stays valid; only active flag flips |
| FormEditorContent coupling to company-settings API surface | All existing callers already skip the legacy path (compat shim handles that); frontend just changes the query key + endpoint |
