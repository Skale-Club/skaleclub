# Plan M3-01-01 — Multi-Forms Schema + Compat Shim

**Milestone:** 3 — Multi-Forms Support
**Phase:** 1 — Schema + Migration + Compat Shim (Phase 5.1 of the detailed plan)
**Status:** In progress
**Reference:** `.planning/phases/05-multi-forms/PLAN.md` (full 5-phase plan)

---

## Goal

Introduce a `forms` table so the app can support N independent lead capture forms instead of the single global `company_settings.formConfig`. Migrate zero-downtime: the legacy endpoints keep working by reading/writing the default form.

After this phase ships, no UI change is visible to users — but the backend is ready for multi-form wiring in Phase 2 (M3-02).

---

## Decisions Locked (recommendations from the draft plan)

1. Forms = new **top-level admin sidebar section** (distinct from Leads) — applies to Phase 2, not this phase.
2. Public URL = `/f/:slug` dedicated route — applies to Phase 3.
3. Chat picks form **globally** in Chat settings — applies to Phase 3.
4. Deleting a form with leads → **soft-delete** (`is_active=false`) — applies to Phase 2.
5. GHL field collisions across forms → allowed with UI hint — applies to Phase 2.

## Pre-conditions

- `company_settings.form_config` may be null or populated (both cases handled by seed)
- Existing `form_leads` may have 0..N rows (backfill handles empty case gracefully)
- No active PAUL loop blocking

## Data Model

```
forms (
  id          SERIAL PRIMARY KEY
  slug        TEXT NOT NULL UNIQUE       -- kebab-case, e.g. "default", "contact-us"
  name        TEXT NOT NULL              -- admin-facing label
  description TEXT
  is_default  BOOLEAN NOT NULL DEFAULT FALSE  -- exactly one form is default (partial unique idx)
  is_active   BOOLEAN NOT NULL DEFAULT TRUE   -- soft-delete support
  config      JSONB NOT NULL              -- { questions, maxScore, thresholds }
  created_at  TIMESTAMP DEFAULT NOW()
  updated_at  TIMESTAMP DEFAULT NOW()
)

form_leads (
  ... existing columns ...
  form_id     INTEGER REFERENCES forms(id)  -- nullable during migration; tightened in M3-05
)
```

---

## Migration Strategy

1. `CREATE TABLE forms` (with unique index on slug, partial unique index for `is_default=TRUE`)
2. Seed the existing `form_config` from `company_settings` as the default form (slug="default", is_default=true)
3. `ALTER TABLE form_leads ADD COLUMN form_id INTEGER REFERENCES forms(id)` (nullable)
4. Backfill `form_leads.form_id` to the default form's id
5. Leave `form_id` nullable for now — tightened to NOT NULL in M3-05 after all entry points set it explicitly

The migration is **idempotent** (uses `IF NOT EXISTS` everywhere and guarded INSERT). Safe to re-run.

## Compat Shim

| Legacy endpoint | New behavior |
|---|---|
| `GET /api/form-config` | Returns `defaultForm.config` |
| `PUT /api/form-config` | Updates `defaultForm.config` |
| `POST /api/form-leads/progress` | Stores lead with `formId = defaultForm.id` |

The chat AI tool `get_form_config()` continues to resolve to the default form until Phase 3 wires conversation-scoped form selection.

## Risk Log

| Risk | Mitigation |
|---|---|
| Customer with `form_config = NULL` in company_settings | Migration skips seed; app code provisions default form from `DEFAULT_FORM_CONFIG` on first access (via `ensureDefaultForm` storage method) |
| Concurrent default-form switches in Phase 2 | Partial unique index (`WHERE is_default = TRUE`) enforces single default at DB level |
| Orphaned form_leads if a form is hard-deleted | Phase 2 uses soft-delete; hard-delete blocked if leads exist |
| Migration run twice | All DDL is `IF NOT EXISTS`; seed uses `NOT EXISTS` guard |
