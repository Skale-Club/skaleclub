<objective>
## Goal
Drop the legacy `company_settings.form_config` column from Postgres, remove the matching `formConfig` field from the Drizzle + Zod schemas, and delete the two remaining storage fallback branches that read it. Apply the migration to the remote Supabase instance so code and DB are back in sync.

## Purpose
Closes the multi-forms migration: 05-01 already removed every runtime **read** of `company_settings.formConfig` from client + server **request handlers**. The only lingering consumers are two storage helpers (`ensureDefaultForm` seed, `upsertFormLeadProgress` fallback). Removing them together with the column makes `forms` the exclusive source of truth for form config. One-way change — guarded by a human-gated remote apply step.

## Output
- Drizzle Kit migration `migrations/0030_drop_company_settings_form_config.sql` drops the column
- Mirror Supabase CLI migration at `supabase/migrations/20260415120000_drop_company_settings_form_config.sql`
- `shared/schema/settings.ts` has no `formConfig` field on the Drizzle table or the insert Zod schema; unused `FormConfig` import removed
- `server/storage.ts` `ensureDefaultForm` seeds from `DEFAULT_FORM_CONFIG` only; `upsertFormLeadProgress` resolves config from the `forms` row only (no settings fallback)
- Remote Supabase has the column removed and responds to a smoke query with zero rows in `information_schema.columns` for `company_settings.form_config`
</objective>

- Any route handler — all read paths were retired in 05-01.

## SCOPE LIMITS
- Only `company_settings.form_config` is dropped. No other columns, no other tables.
- No deprecation window — the column is dropped outright. Safe because 05-01 already removed every runtime read from handlers, and Task 2 removes the last two storage reads together with the schema field, keeping a same-commit atomic semantic in the code.
- No test additions (project has no test framework per CLAUDE.md).
- No `FormConfig` type changes anywhere; the type survives (forms.config still uses it).
- No Supabase RLS changes (column drop is covered by the owning table's existing policies).
- Do not run `drizzle-kit generate` — we hand-write the SQL to keep migration numbering stable.

## RISK NOTES
- **One-way operation:** the column drop cannot be rolled back without data loss. Data is preserved in `forms.config` of the default form row (seeded from `formConfig` at M3-01 bootstrap), so the practical rollback is `ADD COLUMN form_config jsonb` + leaving it null.
- **Timing:** Task 2 (code) and Task 4 (DB) must ship close together. If code deploys before DB apply, the app runs fine (no reads of the column remain). If DB applies before code deploys, `CompanySettings.formConfig` type would still exist in client bundles but TypeScript-never; no runtime impact because no code path accesses it. Either order is safe — but apply to DB within the same session as the code commit to avoid confusion.
