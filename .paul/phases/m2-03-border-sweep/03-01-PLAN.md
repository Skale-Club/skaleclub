**Milestone:** 2 — Design + Structural Refactor

## Goal

Eliminate every hardcoded harsh border color (`border-slate-*`, `border-gray-*`, etc.) from the application code. Replace with the soft alpha token (`border` resolving to `--border`) OR remove the border entirely when redundant with surrounding visual structure.

Phase 2 cleaned the shadcn base components. Phase 3 cleans the 19 application-layer files that still force harsh borders, bypassing the token.

## Decision Tree (per occurrence)

For each `border-<color>-<shade>` match, apply this tree:

**1. Does the border separate two visually distinct content areas?**
   - **Yes** → Convert: remove the color class, keep `border` (uses token)
   - **No** → Go to 2

**2. Does the element already have `bg-*` + `shadow` that distinguishes it?**
   - **Yes** → Remove the border entirely (redundant)
   - **No** → Convert: keep `border`, use token

**3. Is the border purely ornamental (not structural)?**
   - **Yes** → Remove
   - **No** → Convert

**4. Special cases:**
   - Navbar pill (`border-white/10`) → Intentional, KEEP as-is (it's alpha already, just not via token)
   - Status badges with colored borders (e.g., `border-green-500`) → KEEP if semantic color, they communicate state
   - Form validation states (error border red) → KEEP, semantic

## Risks

| Risk | Mitigation |
|------|-----------|
| Over-removing borders → loss of visual separation | Apply decision tree strictly; when in doubt, convert (not remove) |
| Semantic color borders (red error, green success) lost | Exception rule in decision tree — keep semantic colors |
| Navbar pill loses intentional alpha | T3 separate, verify before change |
| Dark mode low contrast after converting | Post-check visually in dark; bump alpha to 10% if needed |
