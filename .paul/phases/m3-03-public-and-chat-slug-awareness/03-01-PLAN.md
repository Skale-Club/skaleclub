# Plan M3-03-01 — Public Form + Chat Widget Slug Awareness

**Milestone:** 3 — Multi-Forms Support
**Phase:** 3 — Public Form + Chat Widget Slug Awareness (Phase 5.3 of the detailed plan)
**Status:** Ready for apply
**Reference:** `.planning/phases/05-multi-forms/PLAN.md`
**Depends on:** M3-02 (admin CRUD UI + public `GET /api/forms/slug/:slug/config` endpoint)

---

## Goal

Route each public-facing capture point to a specific form:

- **Public link** — visiting `/f/:slug` opens a form modal seeded with that form's config, and lead submissions are stamped with that form's `form_id`.
- **Chat widget** — admins pick which form the AI assistant qualifies with, via a new selector in Chat settings. Default stays as today (the form with `is_default=true`).

After this phase ships, a company can publish two distinct lead capture URLs that store into the same `form_leads` table but through different form configs and with distinct `form_id` stamps.

---

## Scope

**In:**

*Backend*
- `chat_settings.form_slug` — new nullable text column. When null, chat uses the default form (current behavior).
- `POST /api/forms/slug/:slug/leads/progress` — new public endpoint. Resolves slug → `form`, calls `upsertFormLeadProgress` with `formId` in metadata. Keeps `/api/form-leads/progress` alive as the legacy default-form path.
- `get_form_config`, `save_lead_answer`, `get_lead_state`, complete-lead GHL sync — all resolve via new `resolveChatForm()` helper (reads `chatSettings.formSlug`, falls back to `ensureDefaultForm()`).

*Frontend*
- New route `/f/:slug` rendering a lightweight `<PublicForm>` page: hero-like wrapper that auto-opens `<LeadFormModal>` with `formSlug` set; shows 404 state if slug unknown.
- `<LeadFormModal>` accepts `formSlug?: string` prop:
  - Swaps config fetch to `/api/forms/slug/:slug/config`.
  - Swaps submission to `/api/forms/slug/:slug/leads/progress`.
  - No prop → legacy `/api/form-config` + `/api/form-leads/progress` (preserves current Home modal behavior).
- Admin Chat section — new "Qualification Form" selector. Pulls from `/api/forms` (active only). Default option shown when `formSlug` is null.

*Schema*
- Drizzle + Supabase CLI migration: add `chat_settings.form_slug TEXT` (nullable).

*Translations*
- PT strings for every new user-facing string (per memory rule).

**Out (deferred):**

- Leads section filter by form (M3-04).
- Dashboard scoped by form (M3-04).
- Removing legacy endpoints and `company_settings.form_config` column (M3-05).
- Per-conversation form selection in chat (future — current decision is global selection).
- GHL field mapping UI hints for multi-form conflicts (deferred, allowed silently).

## API Contracts

### `POST /api/forms/slug/:slug/leads/progress`

```
URL:    /api/forms/:slug/leads/progress
Auth:   public (no requireAdmin)
Body:   same shape as existing /api/form-leads/progress
200:    { ...FormLead }
404:    Slug unknown or form inactive
400:    Validation error
```

Server resolves `slug → form`, then calls `storage.upsertFormLeadProgress(payload, { userAgent, formId: form.id, source: 'form' }, form.config)`. GHL sync runs identically using the form's own `ghlFieldId`s.

### Legacy `POST /api/form-leads/progress`

Unchanged — continues routing to the default form (via compat shim from M3-01). Phase 5 removes this endpoint.

### `resolveChatForm()` (server-side helper)

```ts
async function resolveChatForm(): Promise<Form> {
  const chat = await storage.getChatSettings();
  if (chat?.formSlug) {
    const f = await storage.getFormBySlug(chat.formSlug);
    if (f && f.isActive) return f;
  }
  return storage.ensureDefaultForm();
}
```

Used by all 4 chat tool sites + `getOrCreateLeadForConversation`. No behavior change when `formSlug` is null (delegates to `ensureDefaultForm`).

## UI Design

### `/f/:slug` page (`<PublicForm>`)

Minimal wrapper:
- Fetches `/api/forms/slug/:slug/config` on mount
- If 404 → shows "Form not found" card with a link back to `/`
- Otherwise renders `<LeadFormModal open formSlug={slug} onClose={() => setLocation('/')} />`

The modal handles everything else. No marketing surround — the URL is intended for direct-share (QR code, email link, etc.).

### Admin Chat settings — form selector

New card in the existing Chat settings layout:

```
┌ Qualification Form ────────────────────────── ┐
│ The AI assistant will qualify leads using:    │
│                                                │
│ Form: [ Default Form (default) ▼ ]            │
│       ↳ Changes take effect on new conversations│
└────────────────────────────────────────────────┘
```

Select options: all `isActive=true` forms, plus a first item "Use default form" mapping to `formSlug = null`.

## Risk Log

| Risk | Mitigation |
|---|---|
| Admin picks a form, then archives it | `resolveChatForm` checks `isActive`; falls back to default silently. Consider adding a UI warning in M3-02 if user tries to archive a form referenced by Chat settings — deferred, not blocking. |
| Public submits to a slug that was archived between config fetch and submit | Public endpoint checks `isActive`; returns 404. Client shows "Form not found" toast and lets user retry. |
| Chat conversations in progress when slug changes | Decision locked: global chat form selection; new conversations use the new form. Existing conversations keep their current form because `getOrCreateLeadForConversation` reads the slug on the first message, not every call. Document in UI copy ("Changes take effect on new conversations"). |
| Legacy `/api/form-leads/progress` coexistence | Both endpoints call the same storage upsert; compat shim from M3-01 guarantees consistency until M3-05. |
| LeadFormModal is large and stateful (~700 LOC) | Only two lines change (config fetch + submit endpoint); prop is additive. |
